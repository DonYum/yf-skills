# Generate Monorepo Docs

为 monorepo 多应用系统生成系统级文档体系，整合所有 submodule 已有文档。

**Announce at start:** "I'm using the generate-monorepo-docs skill 来为这个 monorepo 生成系统级文档。"

**前置条件：** 所有 submodule 应用的 docs 已生成（推荐先用 `generate-legacy-project-docs` 为每个 submodule 生成文档）。本 skill 只读取 md 文档，不直接读取代码。

---

## 执行流程（严格按顺序，不可跳过）

### Phase 1: 读取与分析

#### 1.1 读取 monorepo 根文档

按以下顺序读取，存在就读：

```
AGENTS.md
CLAUDE.md
AGENT.md
README.md
docs/**/*.md
doc/**/*.md
*.md
```

记录已有内容，后续禁止丢失已有信息。

#### 1.2 扫描所有 submodule

```bash
# 优先使用 Git submodule 元信息
if [ -f .gitmodules ]; then
  git config --file .gitmodules --get-regexp path | awk '{print $2}' | sort
else
  # 回退：启发式识别候选模块目录（输出目录而不是文件路径）
  find . -maxdepth 3 -type f \( -name "CLAUDE.md" -o -name "AGENTS.md" -o -name "AGENT.md" -o -name "README.md" \) -exec dirname {} \; | sort -u
fi
```

若使用回退识别方式，必须在最终摘要中标注"候选模块可能包含非 submodule 目录"并给出人工复核建议。

#### 1.3 逐个读取 submodule 文档

对每个 submodule 读取：

- `AGENTS.md` / `CLAUDE.md`（提取职责、接口契约、禁止修改区域、高风险区域）
- `README.md`（提取技术栈、启动方式、测试方式）
- `docs/` 下所有 md 文件（提取架构、跨应用规则、依赖、变更影响、已知问题）

#### 1.4 形成系统级分析

读完所有 submodule 文档后，明确以下问题的答案再进入 Phase 2：

1. 整个系统负责什么？各 submodule 分别负责什么？
2. submodule 之间如何调用？调用链是什么？
3. 哪些接口是跨模块契约，不可随意修改？
4. 哪些模块是高风险的？
5. 修改某个 submodule 会影响哪些其他 submodule？
6. 系统级的已知问题有哪些？
7. 统一的构建/测试/部署命令是什么？

---

### Phase 2: 生成文档

按以下顺序生成。

#### 2.0 根文档命名策略（必须先确定）

- 优先级固定为：`AGENTS.md` > `CLAUDE.md`
- 若二者都存在，只更新优先级最高的文件，其他文件保留为跳转或兼容入口
- 若都不存在，默认创建 `AGENTS.md`
- 根目录必须保持**单一 AI 主入口文档**

## 文件 2: AGENTS.md 兼容入口（仅当主根文档不是 AGENTS.md 时）

放在 monorepo 根目录。仅包含跳转，不维护重复内容。

```markdown
# AGENTS.md（兼容入口）
本仓库 AI 主入口文档为 `./CLAUDE.md`。
请跳转至该文件获取完整规则与索引。
```

若主根文档已经是 `AGENTS.md`，则跳过本文件，不重复生成。

## 文档互链规则（必须遵守）

- 系统级主根文档（AGENTS.md / CLAUDE.md）必须包含完整文档索引（系统级 + 所有 submodule）
- AGENTS.md 仅在作为主根文档或兼容入口时存在；作为兼容入口时只保留跳转信息
- README.md 必须包含文档导航区
- docs/ 内文档引用其他文档必须使用相对路径
- 所有链接必须实际可达，不要链接不存在的文件

---

## 强约束规则

1. **禁止空泛描述**——写不出具体内容就不写该区块
2. **禁止无依据推测**——不能凭猜测写结论；证据不足时必须标注"信息不足/待确认"并给出验证路径
3. **必须明确边界**——每个模块负责什么、不负责什么
4. **必须明确风险**——高风险模块和禁止修改区域
5. **必须明确禁止跨模块操作**——哪些操作不允许跨模块执行
6. **最小变更原则**——已有文档只补充和重组，不随意删改
7. **禁止删除人工补充区**——`## 人工补充约束` 区块内容永远保留
8. **不涉及的文档不生成**——不要生成空文件或只有标题的文件

## 成功标准

生成的文档体系必须能让 AI 准确回答：

1. 修改某模块会影响哪些应用？
2. 哪些区域禁止修改？
3. 系统跨模块依赖关系是什么？
4. 测试入口和最小验证集在哪里？
5. 某个需求应该路由到哪个模块？

如果文档无法支撑回答以上任一问题，视为生成失败，需要补充。

## Common Mistakes

**没有读完所有 submodule 文档就开始写**
- **Problem:** 系统级文档遗漏关键模块的信息
- **Fix:** Phase 1 必须遍历所有 submodule 的 md 文档

**直接读代码而非读文档**
- **Problem:** 本 skill 定位是整合已有文档，不是代码分析
- **Fix:** 只读 md 文件，如果 submodule 缺少文档，提示用户先用 generate-legacy-project-docs 生成

**跨模块依赖关系不完整**
- **Problem:** 遗漏了某些模块间的调用关系
- **Fix:** 交叉验证每个 submodule 的 05_cross_app_rules.md

**生成了不涉及的文档**
- **Problem:** docs/ 下出现只有标题没内容的空文件
- **Fix:** 不涉及的文档直接不生成

**破坏了已有人工约束**
- **Problem:** 覆盖了 `## 人工补充约束` 区块内容
- **Fix:** 读取时记录该区块内容，生成时原样保留
