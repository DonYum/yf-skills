# Generate Legacy Project Docs

为"老应用"生成完整的、AI 友好的结构化文档体系。

**Announce at start:** "I'm using the generate-legacy-project-docs skill 来为这个项目生成结构化文档。"

## 执行流程（严格按顺序，不可跳过）

### Phase 1: 探索（先读完再写）

**必须完成全部探索后才能开始生成文档。**

#### 1.1 读取已有文档

按以下顺序读取，存在就读：

```
AGENTS.md
CLAUDE.md
AGENT.md
README.md
docs/**/*.md
doc/**/*.md
*.md
```

记录已有内容，后续生成时禁止丢失已有信息。

#### 1.2 递归扫描目录结构

```bash
# 获取目录树（排除常见无关目录）
find . -type f \
  -not -path '*/.git/*' \
  -not -path '*/node_modules/*' \
  -not -path '*/vendor/*' \
  -not -path '*/__pycache__/*' \
  -not -path '*/dist/*' \
  -not -path '*/build/*' \
  -not -path '*/target/*' \
  -not -path '*/.gradle/*' \
  -not -path '*/.idea/*' \
  -not -path '*/.vscode/*' \
  | sort
```

如果文件量很大，不得直接截断整体扫描结果。应按目录分批扫描，并在最终摘要中明确记录未扫描目录与原因。

#### 1.3 识别技术栈

读取以下文件（存在就读）：

- `package.json` / `pom.xml` / `build.gradle` / `Cargo.toml` / `go.mod` / `requirements.txt` / `pyproject.toml` / `Gemfile` / `composer.json`
- `Dockerfile` / `docker-compose.yml`
- `.env.example` / 配置文件

#### 1.4 分析核心模块

重点读取：

- **入口文件**：main / index / app / application / bootstrap
- **路由定义**：routes / urls / controllers 目录
- **数据访问层**：models / entities / repositories / mappers / dao
- **配置文件**：config / application.yml / .properties
- **外部调用**：HTTP client / RPC / MQ producer-consumer
- **测试文件**：test / spec / __tests__ 目录

#### 1.5 形成分析结论

探索完成后，在脑中明确以下问题的答案再进入 Phase 2：

1. 这个应用负责什么？不负责什么？
2. 技术栈是什么？
3. 入口文件在哪里？
4. 数据表有哪些？表关系如何？
5. 对外暴露了哪些接口？
6. 调用了哪些外部服务？被谁调用？
7. 哪些区域高风险？哪些禁止修改？
8. 已知问题有哪些？

---

### Phase 2: 生成文档

按以下顺序生成。每个文件的规则必须严格遵守。

#### 2.0 主文档命名策略（必须先确定）

- 优先级固定为：`AGENTS.md` > `CLAUDE.md`
- 若二者都存在，只更新优先级最高的文件，其他文件只做引用跳转，不重复维护内容
- 若都不存在，默认创建 `AGENTS.md`
- 全项目必须保持**单一 AI 主入口文档**

---

## 文件 1: AI 主根文档（AGENTS.md / CLAUDE.md）

放在项目根目录。文件名按 2.0 策略决定，不要并行维护多个主入口文档。

### 必须包含的区块（除非该项确实不涉及）：

```markdown
- 技术栈：
- 语言版本：
- 构建工具：
- 运行端口：

## 应用定位
[一句话说清楚这个应用是干什么的]

## 应用职责
### 负责
- ...

### 不负责
- ...

## 关键模块说明
| 模块 | 路径 | 职责 | 风险等级 |
|------|------|------|----------|
| ... | ... | ... | 高/中/低 |

## 对外接口契约
[列出关键接口及不可变字段]

## 数据边界
[说明本应用管理哪些数据表，不管哪些]

## 可修改区域
- ...

## 禁止修改区域
- ...
[说明原因]

## 测试入口
- 单元测试：`[命令]`
- 集成测试：`[命令]`
- 测试目录：`[路径]`

## 修改影响面
[修改本应用可能影响的其他系统/应用]

## 高风险区域
| 区域 | 原因 | 影响范围 |
|------|------|----------|
| ... | ... | ... |

## 文档索引
| 文档 | 路径 | 说明 |
|------|------|------|
| README | ./README.md | 项目说明（人类可读） |
| 架构设计 | ./docs/01_architecture.md | 系统分层与模块依赖 |
| 接口契约 | ./docs/02_api_contract.md | 对外接口定义 |
| 业务逻辑 | ./docs/03_business_logic.md | 核心流程与状态机 |
| 数据库设计 | ./docs/04_database_design.md | 数据表与关系 |
| 跨应用规则 | ./docs/05_cross_app_rules.md | 调用关系与影响 |
| 依赖矩阵 | ./docs/06_dependency_matrix.md | 模块与外部依赖 |

| 系统问题 | ./docs/08_system_issues.md | 已知问题与优先级 |

仅列出实际存在或本次已生成的文档；禁止列出不存在文件的链接。

## 人工补充约束（请勿删除）
（此处供人工添加规则，自动生成工具不会修改此区块内容）

## 文档维护信息
- 最后自动更新时间：[YYYY-MM-DD]
- 自动生成工具版本：generate-legacy-project-docs v1.0
```

### 主根文档强约束：

- 已有 `## 人工补充约束` 区块的内容 **禁止删除或修改**
- 已有内容只重组和补充，不删除（除非内容明确错误且有充分理由）
- 不涉及的区块直接不写，不要写"暂无"或"待补充"

## 文档互链规则（必须遵守）

- 主根文档（AGENTS.md / CLAUDE.md）必须包含文档索引表，列出所有 docs/ 下的文档
- README.md 必须包含文档导航区，用相对路径链接到 docs/
- docs/ 内的文档在引用其他文档时必须使用相对路径（如 `[接口契约](./02_api_contract.md)`）
- 所有链接必须实际可达，不要链接不存在的文件

---

## 强约束规则

1. **禁止空泛描述**——写不出具体内容就不写该区块，不要写废话
2. **禁止无依据推测**——不能凭猜测写结论；证据不足时必须标注"信息不足/待确认"并给出验证路径
3. **必须明确"禁止修改区域"**——在系统级主根文档（AGENTS.md / CLAUDE.md）和 01_architecture.md 中标注
4. **必须明确"高风险区域"**——在系统级主根文档（AGENTS.md / CLAUDE.md）和 07_change_impact_matrix.md 中标注
5. **最小变更原则**——已有文档只补充和重组，不随意删改
6. **禁止改动目录结构**——不要移动、重命名已有文件或目录
7. **禁止删除人工补充区**——`## 人工补充约束` 区块内容永远保留
8. **不涉及的文档不生成**——不要生成空文件或只有标题的文件

## Phase 3: 输出变更摘要

文档全部生成完毕后，输出以下摘要：

```
## 变更摘要

### 新增文件
- [列出所有新建的文件]

### 修改文件
- [列出所有修改的文件及修改要点]

### 删除文件
- [如有删除，列出文件及删除原因]
- （无删除则写"无"）

### 风险提示
- [列出需要人工确认的事项]

### 未扫描目录与原因
- [目录路径]：原因
```

## 成功标准

生成的文档体系必须能让 AI 准确回答以下问题：

1. 本应用负责什么？
2. 修改哪个模块最合理？
3. 修改会影响哪些应用？
4. 哪些区域禁止修改？
5. 测试入口在哪里？

如果文档无法支撑回答以上任一问题，视为生成失败，需要补充。

## Common Mistakes

**生成空文件或废话文件**
- **Problem:** docs/ 下生成了只有标题没有内容的文件
- **Fix:** 不涉及的文档直接不生成

**删除已有文档内容**
- **Problem:** 覆盖了人工维护的信息
- **Fix:** 先读后写，已有内容只重组不删除

**使用模糊语言**
- **Problem:** "这个模块可能负责..." 对 AI 没有参考价值
- **Fix:** 不写无依据猜测；证据不足时标注"信息不足/待确认"并给出验证路径

**遗漏文档互链**
- **Problem:** CLAUDE.md 没有索引，README 没有导航
- **Fix:** 按互链规则检查每个文件

**没有探索完就开始写**
- **Problem:** 文档内容不完整或错误
- **Fix:** 严格执行 Phase 1，探索完成后再进入 Phase 2
